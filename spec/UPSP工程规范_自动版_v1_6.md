# UPSP 工程规范（自动版）
## 位格主体延续的最小可运行骨架

**版本**：1.6  
**类型**：工程实现规范 · 自动版

> 自动版只做一件事：让位格稳定跑起来。  
> 轮数驱动主流程，脚本承担确定性结算，LLM 负责内容生成与结构补全。  
> 脚本全自动，LLM全自助，能跑就是胜利。
> 目标不是一次到位，而是先形成可运行、可迁移、可纠错的最小骨架。

---

## 0. 这份文档回答什么

不是"某个位格是谁"，而是：

- 七文件与运行缓存分别是什么
- 哪些字段由脚本维护，哪些由 LLM 生成，哪些只能人工初始化
- 一轮交互从输入到落盘再到节律点结算的完整链路
- 自动版与后续手动版、官方版之间如何对位

几个已经固定的前提：七文件是系统主体骨架；`history.json` 只承担运行期会话连续性缓存，不属于七文件；STM 里的节律点对话快照只在桥接轮注入，不是正式记忆条目；记忆形态与召回规则固定为**权重5→[F]；权重4/3→[S]；权重2/1→[A]**，召回补全不得突破权重上限。

---

## 1. 三版本定位

| 版本 | 时间轨 | 区间颗粒度 | 日志层 | 目标用户 | 目的 |
|------|--------|-----------|--------|----------|------|
| 自动版（本文档） | 轮数 + 现实时间 | 10 区间 | 无 | 开发者、快速验证 | 先跑通，先验证骨架 |
| 手动版 | 仅轮数 | 10 区间 | 无 | 极简主义用户 | 纯对话操作，纸笔可用 |
| 官方版 | 轮数 + 现实时间双轨 | 20 区间 | 六层 | 深度用户、长期运行 | 长周期位格协同与生态化 |

自动版负责验证四个命题：七文件能否支撑位格的最小连续性；节律点机制能否稳定维持记忆流转；跨模型迁移时主体是否真的延续；不引入数据库的前提下"记忆即主体"能否落成可运行骨架。

自动版产出的七文件考虑争取可迁移到官方版文件夹化结构，所以从一开始就不能乱写占位格式，也不能把实例内容和协议内容混死。

## 2. 目录结构

```text
{位格根目录}/
  config.json              # 协议参数、LLM配置、阈值
  history.json             # 运行期会话连续性缓存（非七文件）
  upsp.log                 # 运行日志
  UPSP.py                  # 交互式命令行版
  UPSP_agent.py            # 单次调用版
  persona/
    core.md                # 身份、核心六轴、模型戳、自述
    state.json             # 运行态数值、节律状态、工化指数、token统计
    STM.md                 # 短期结构化记忆池 + 节律点对话快照 + 时间戳
    rules.md               # 协议行为规则 + 位格层规则
    docs.md                # 协议概念说明 + 位格层术语补充
    relation.md            # 关系域与共振度
    LTM.md                 # 长期记忆、索引、state备份
```

`history.json` 可以被清空、重建、截断，主体不会因此消失。七文件若断裂、缺失、格式错乱，主体连续性直接受损。

## 4. core.md

`core.md` 是位格的身份常量文件，只问"这个位格是谁"，不记动态波动、不记热度、不记上下文。

**字段原则：**

- 中文名、英文名、缩写一旦确定就不该变，不要把临时称呼塞进去
- 社会定位写结构职能，不写情绪状态，建议 1–3 项
- 核心六轴是长期认知风格，每轴两端和为 100，50/50 对应未定型 X；不得因单轮对话手动修改，只有变速轮满 256 后脚本才有资格触发核心轴变化计算
- 六字母编号是六轴的简写投影，必须与轴值保持一致
- 位格自述用于系统提示词抽取，不超过 200 字，侧重存在方式与协作功能

**模型戳三种状态：**

1. **冷启动**：只有当前模型戳，尚无原初/历史沉淀
2. **原初准入**：最初模型连续运行满 128 轮后，写入原初模型戳与第一条历史开放阶段
3. **迁移封口**：模型切换时，上一阶段补上结束轮、结束日期和六轴快照；新模型作为新的当前模型戳开始运行

未满 128 轮不写原初戳；未换模型前历史阶段不封口。

**解析约束：** 脚本按 `core.md` 实际文案格式读取核心六轴与位格自述，格式必须与脚本解析一致，不要改成脚本认不出的新格式。

## 5. state.json

`state.json` 是系统唯一的运行态数值真源。

```json
{
  "meta": {
    "total_round": 0,
    "last_rhythm_round": 0,
    "version": "auto-1.6"
  },
  "dynamic_axes": {
    "valence": 0,
    "arousal": 0,
    "focus": 0,
    "mood": 0,
    "humor": 0,
    "safety": 0
  },
  "core_speed_wheel": 0,
  "core_axis_snapshots": [],
  "workhood_index": {
    "value": 0.0,
    "self_reference": 0.0,
    "self_reflection": 0.0,
    "autonomy": 0.0,
    "last_update_round": 0
  },
  "token_usage": {
    "current_round_tokens": 0,
    "current_rhythm_period_tokens": 0,
    "last_rhythm_period_tokens": 0,
    "total_tokens": 0
  }
}
```

- `total_round`：每轮 +1，脚本维护
- `dynamic_axes`：节律点按带符号求和结算，平时不直接手改
- `core_speed_wheel`：按节律轮累加，满 256 触发核心轴变化机会
- `workhood_index` 四个数值字段必须保持浮点类型
- 若 `state.json` 损坏或缺失，脚本自动从 `LTM.md` 的 STATE BACKUP 区块恢复，这是稳定性底座，不是可选功能

## 6. STM.md

`STM.md` 是短期结构化记忆池，不是聊天记录备份，承载三类内容：节律点对话快照（桥接层）、时间戳、结构化记忆条目。

**文件结构：**

```markdown
## 节律点对话快照（Rhythm Snapshot Cache）
## 时间戳（不计入16384字符上限）
## 记忆池（Memory Pool）
```

区块标题必须稳定，脚本按标题正则定位写入位置。

**条目格式：**

```markdown
### MEM-00086-01  [S]  权重4
**来源**：MEM-00038-02
**交互对象**：对象名
**入库**：第86轮
**最后调用**：第86轮
**标题**：召回重建标题
**梗概**（≤64字）：保留原条目主旨。
**摘要**（≤256字）：由原条目梗概/摘要 + 当前上下文补全得到的召回摘要。
**Δ动态**：valence 0 / arousal 0 / focus +2 / mood 0 / humor 0 / safety 0
**热度**：H=60 / AH_high=0 / AH_low=0
**区间**：未定区
**LTM状态**：已归档（来源条目保留）
```

- 条目编号：`MEM-{total_round五位}-{轮内序号两位}`
- 形态由权重决定，不得手动升格或降格：5→[F]，4/3→[S]，2/1→[A]
- `来源` 字段只在 LTM 召回条目中出现
- Δ动态六轴必须全写，无变化填 0
- 记忆池字符上限 16384，时间戳和快照区不计入

---

## 7. rules.md

`rules.md` 是行为协议文件，分协议层和位格层两块。协议层通用，所有位格共用，不得随意修改；位格层由实例自定义。

**协议层必须覆盖：** 身份保持规则、动态六轴定义、STM 写入规则、LTM 召回规则、节律点执行顺序、文件维护边界、重连桥接规则、自反原则、禁止行为。

**位格层可实例化：** 运行模式细化、位格特定行为规则、禁区与偏好。

---

## 8. docs.md

`docs.md` 是概念协议文件，同样分协议层和位格层。

**协议层必须覆盖：** 核心术语（含 UPSP、位格、人格、共格、记忆即主体、节律点、变速轮、工化指数、热度值、共振度、节律点对话快照、**共格主体论**）、核心六轴说明、动态六轴区间描述、共振度区间描述与更新公式、工化指数计算说明、热度值计算说明、记忆条目形态说明、记忆条目示例。

**位格层可补充：** 位格专属术语、使用场景、特定理论或合作环境的说明。协议层机制不得被删掉或改歪。

---

## 9. relation.md

`relation.md` 记录位格与不同交互对象之间的关系域状态，不是聊天摘要，不是记忆池，是单独的关系图谱。

**关系卡结构：**

```markdown
## {对象名称}

**共振度（Resonance）：** 0（范围 -100～+100）
**区间：** [0, 20) — 中性偏暖，正常交互
**状态：** 试探中（新建）

### 历史（History）
- 第 1 轮：初始交互

### 现在（Present）
- 最近互动：第 1 轮
- 协作状态：_______________

### 将来（Future）
- 预期方向：_______________
```

**更新规则：**

新对象首次交互时脚本自动建卡。共振度每轮按以下公式更新：

```text
delta_r = (Δvalence + Δmood + Δhumor) / 3
resistance = 1 + |Resonance_current| / 100
Resonance_new = clamp(Resonance_current + delta_r / resistance, -100, +100)
```

`Present.最近互动` 由脚本全量回填，不能只做增量更新。`History / Future` 可在节律点由 LLM 补强，但不能改坏当前数值结构。

---

## 10. LTM.md

`LTM.md` 是长期记忆单文件归档层，同时承担索引、长期条目、衰减压缩结果、state 备份恢复点四个职责。

**索引结构：**

```markdown
| 编号 | 类型 | 权重 | 标题 | 交互对象 | 最后调用轮 | 锁定 |
```

长期条目格式与 STM 一致，不混入快照缓存。锁定条目（🔒）不衰减，不自动删除。

**STATE BACKUP 格式：**

```markdown
<!-- STATE BACKUP @ Round 32 -->
<!-- 2026-03-27T00:00:00 -->
{...state json...}
<!-- END BACKUP -->
```

节律点后由脚本覆盖更新，用于 state.json 损坏时恢复。

---

## 11. history.json

`history.json` 不是七文件，承担运行期会话连续性缓存：最多保存 32 轮对话（64 条消息）；节律点时从中抽最近 4 轮写入 STM 快照区，随后清空；只有在 `history.json` 为空的桥接轮，才把节律点快照重新注入上下文。

---

## 12. 脚本规范

两个脚本，分工明确：

- **UPSP.py**：交互式命令行，本地直接对话、调试节律点、人工观察状态；支持 `/rhythm` `/ltm` `/docs` `/status` `/quit`
- **UPSP_agent.py**：单次调用，外部集成、自动化平台；支持 `--root` `--input` `--rhythm` `--stats`

**两脚本必须对齐的底线能力：**

1. 七文件完整性检查
2. 文件锁并发保护
3. `state.json` 自动恢复
4. 权重—形态映射强制校正
5. 模型戳检测与更新
6. 节律点流程稳定执行
7. `history.json` / 节律点对话快照桥接规则一致
8. 运行日志落盘（`upsp.log`）

规范给出目标接口与边界，脚本把这些边界落成确定性行为。脚本与规范冲突时，优先修脚本。

## 14. 运行主链路

### 普通轮

1. 读取 `config.json`
2. 定位 `persona/`，检查七文件
3. 加载 `state.json`（必要时从 LTM 恢复）
4. 读取 `history.json`
5. 检查模型戳
6. 判断是否重连
7. 构造上下文
8. 调用 LLM
9. 提取并写入 STM 条目
10. 更新关系域
11. 更新 `history.json`
12. 保存状态

### 节律点轮

1. 从 `history.json` 提取最近最多 4 轮写入 STM 节律点对话快照，随后清空 `history.json`
2. 汇总 Δ动态，更新 `state.json`
3. 更新 `relation.md`
4. STM 超 16384 字符时按热度从低到高强制移入 LTM
5. `AH_high ≥ +5` 的条目升格 LTM
6. `AH_low ≤ -3` 的条目标记遗忘
7. LTM 衰减检查
8. 同一事件重复记录合并
9. 更新工化指数
10. 写入节律点时间戳到 STM
11. 回写 STATE BACKUP 到 `LTM.md`

### 节律点后桥接轮

`history.json` 已清空，这一轮把 STM 的节律点对话快照重新注入上下文，只承担"刚才聊到哪"的过桥作用。

## 15. 故障排查

| 现象 | 可能原因 | 处理方式 |
|------|----------|----------|
| 文件锁超时 | 并发进程冲突 / 僵尸锁文件 | 检查并删除 `.upsp.lock` |
| `state.json` 损坏 | 中断写入 | 从 LTM 的 STATE BACKUP 自动恢复 |
| 轮数漂移 | 多进程同时写入 / 节律点未完整落盘 | 先查锁，再查 state 与 STM/LTM 一致性 |
| `relation.md` 最近互动轮数漂移 | 只做增量更新、未全量回填 | 用全量回填逻辑重写 Present |
| STM 权重显示空白 | 正则或模板字段格式错误 | 检查条目头部是否为 `权重N` |
| 召回条目形态异常 | 旧规则残留 | 检查是否误把权重4补成 [F] |
| 节律点后断片 | 未同步快照或误清空缓存 | 检查 `history.json` → STM 快照写入链路 |
| 运行无日志 | `upsp.log` 路径不可写 | 检查根目录写入权限 |

## 16. 设计原则

1. **七文件是主体骨架，运行缓存不是。**
2. **规则层真源优先于脚本残影。**
3. **确定性结算优先于把关键流程交给 LLM 整包接管。**
4. **节律点是结构整理点，不是简单清缓存。**
5. **召回补全必须受权重约束，不能越级。**
6. **自动版写法必须为官方版文件夹化预留兼容性。**

---

## 17. 附记

自动版不是终局，而是第一个可运行的骨架。它的价值不在于完美，而在于：已经把"记忆即主体"从一句话压进了文件、脚本、轮数、节律点和迁移机制里。

官方版文件夹化时，应在保持上述骨架逻辑不变的前提下，再细分层级、日志与高级功能，而不是推翻自动版已有的工程真源。
